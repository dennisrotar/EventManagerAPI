# Event Manager API

Сервис для управления мероприятиями и бронированиями. Базовый каркас приложения, реализованный на ASP.NET Core Web API с применением принципов чистой архитектуры (Clean Architecture), паттернов IRepository и фоновых задач, а также интегрированный с реляционной базой данных PostgreSQL через Entity Framework Core.

## Технологический стек

- **C# / .NET 8**
- **ASP.NET Core Web API**
- **База данных:** PostgreSQL (через Docker)
- **ORM:** Entity Framework Core (с провайдером Npgsql)
- **Фоновая обработка:** BackgroundService (отложенная обработка бронирований)
- **Потокобезопасность:** `SemaphoreSlim`, `lock` (защита от овербукинга)
- **Документация:** Swagger (OpenAPI)
- **Тестирование:** xUnit, EF Core InMemory Provider

## Архитектура проекта

Проект разработан с соблюдением принципов чистой архитектуры (Clean Architecture) и SOLID:

- **Разделение ответственности (SRP):** Логика разделена на контроллеры (маршрутизация), сервисы (бизнес-логика) и контекст данных (доступ к БД).
- **Инверсия зависимостей (DIP):** Сервисы зависят от абстракций (`IEventService`, `IBookingService`) и `AppDbContext`, а не от конкретных реализаций.
- **Жизненные циклы DI:**
    - `AppDbContext` зарегистрирован как **Scoped** (один контекст на HTTP-запрос).
    - Доменные сервисы зарегистрированы как **Scoped**.
    - Фоновый сервис зарегистрирован как **Singleton** (работает вне контекста HTTP-запросов).
- **Обработка ошибок:** Единообразный формат RFC 7807 (Problem Details) для 400, 404, 409 и 500 ошибок через `GlobalExceptionHandler`.
- **База данных:** Маппинг сущностей реализован через Fluent API (`IEntityTypeConfiguration`). Схема БД генерируется автоматически при первом запуске.
- **Observability (Логирование):** Внедрен `ILogger` в контроллеры и сервисы для трассировки жизненного цикла запросов и отладки.

## Подготовка и запуск проекта

### 1. Требование PostgreSQL

Для работы приложения требуется установленный **Docker** (для поднятия контейнера с PostgreSQL).

### 2. Запуск базы данных

В корне проекта находится файл `docker-compose.yml`. Откройте терминал в корне проекта и выполните команду:


```bash
docker compose up -d
```
### 3. Настройка строки подключения

Строка подключения к PostgreSQL находится в `EventManager.Api/appsettings.json`.

```json
{
	"ConnectionStrings": {
		"DefaultConnection": "Host=localhost;Port=5432;Database=eventapi;Username=postgres;Password=postgres"
	}
}
```
### 4. Автоматическое создание схемы БД

В файле `Program.cs` перед стартом API вызывается метод `EnsureCreated()`.


```C#
using (var scope = app.Services.CreateScope())
{

	var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
	
		db.Database.EnsureCreated();

}
```

**Что это значит:** Entity Framework Core при первом запуске проверяет наличие таблиц `events` и `bookings` в PostgreSQL. Если их нет — автоматически создаст их структуру согласно Fluent API конфигурациям. При последующих запусках метод ничего не делает, обеспечивая сохранность данных.
### 5. Сборка и запуск

1. Клонируйте репозиторий и перейдите в папку решения.
2. Выполните сборку:
```bash
    dotnet build
```
3. Запустите API (обратите внимание на флаг `--project`, так как `.csproj` лежит во вложенной папке):
```bash
dotnet run --project EventManager.Api/EventManagerAPI.csproj
``` 
4. Откройте Swagger UI по адресу из логов консоли (например `http://localhost:5000/swagger`).

## Запуск тестов

Для запуска юнит-тестов выполните команду из корня решения:
```bash
dotnet test
```
### Использование InMemory-провайдера в тестах
Так как юнит-тесты должны быть быстрыми и полностью изолированными от внешних зависимостей (включая реальную PostgreSQL), проект тестов использует **Microsoft.EntityFrameworkCore.InMemory**, а не реальную БД.

В тестовых классах DI-контейнер настраивается так:


```C#
var dbName = Guid.NewGuid().ToString(); // Уникальная БД для каждого класса тестов
services.AddDbContext<AppDbContext>(options =>
options.UseInMemoryDatabase(dbName));
```

## Документация API

### События (Events)

|Метод|Эндпоинт|Описание|Успешный статус|
|---|---|---|---|
|`GET`|`/events`|Получить список мероприятий (с фильтрацией и пагинацией)|200 OK|
|`GET`|`/events/{id}`|Получить мероприятие по ID|200 OK / 404 Not Found|
|`POST`|`/events`|Создать новое мероприятие (с обязательным указанием `totalSeats`)|201 Created / 400 Bad Request|
|`PUT`|`/events/{id}`|Обновить мероприятие целиком|204 No Content / 404 Not Found|
|`DELETE`|`/events/{id}`|Удалить мероприятие|204 No Content / 404 Not Found|

### Бронирования (Bookings)
| Метод  | Эндпоинт            | Описание                                             | Успешный статус        |
| ------ | ------------------- | ---------------------------------------------------- | ---------------------- |
| `POST` | `/events/{id}/book` | Создать бронь (быстрый ответ + отложенная обработка) | 202 Accepted           |
| `GET`  | `/bookings/{id}`    | Получить статус брони                                | 200 OK / 404 Not Found |

**Особенности:**

- `POST /events/{id}/book` мгновенно сохраняет бронь со статусом `Pending` и возвращает `202 Accepted`.
- Если мест нет, `POST /events/{id}/book` возвращает **409 Conflict** (защита от овербукинга).

### Пример ответа события

```json
{
    "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "title": "Тренировка по баскету",
    "description": "Опоздаешь-отожмёшься!",
    "startAt": "2024-01-15T20:00:00Z",
    "endAt": "2024-01-15T23:00:00Z",
    "totalSeats": 50,
    "availableSeats": 42
}
```
### Фильтрация и пагинация (GET /events)

Эндпоинт `GET /events` принимает опциональные query-параметры для поиска:

- `title` (string) — поиск по названию (частичное совпадение, регистронезависимый).
- `from` (DateTime) — события, которые начинаются **не раньше** указанной даты.
- `to` (DateTime) — события, которые заканчиваются **не позже** указанной даты.
- `page` (int, по умолчанию 1) — номер страницы (не может быть < 1).
- `pageSize` (int, по умолчанию 10) — количество элементов на странице (не может быть < 1).

## Правила валидации

Сервис осуществляет строгую проверку входных данных:

- **Title**: Не может быть пустой строкой.
- **Даты**: `EndAt` строго позже `StartAt`.
- **Дата в прошлом**: `StartAt` не может быть меньше текущего времени сервера (UTC).
- **Пагинация**: Параметры `page` и `pageSize` не могут быть меньше 1 (возвращает 400 Bad Request).

## Обработка ошибок

Все ошибки (как от встроенного валидатора, так и от бизнес-логики) возвращаются в **единообразном формате RFC 7807 (Problem Details)**.

### Пример 1: Ошибка валидации (400 Bad Request)

Возникает при невалидных данных в теле запроса или некорректных параметрах пагинации.


```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "instance": "/events",
  "traceId": "00-873ba4af2ca7f9c6f861fb3cdb5c6669-0c1015a54df57858-00",
  "errors": {
    "Title": [
      "Название мероприятия не может быть пустым!"
    ]
  }
}
```


### Пример 2: Ресурс не найден (404 Not Found)

Возникает при запросе, обновлении или удалении несуществующего ID.


```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.5",
  "title": "Not Found",
  "status": 404,
  "detail": "Мероприятие с ID 00000000-0000-0000-0000-000000000000 не найдено",
  "instance": "/events/00000000-0000-0000-0000-000000000000",
  "traceId": "00-2f74d953e90ca3678214d2e930e48b51-fd9e4692eb25c93b-00"
}
```
## Конкурентность и защита от овербукинга

### Новые поля модели Event

- `TotalSeats` (int) — общее количество мест на мероприятии.
- `AvailableSeats` (int) — текущее количество свободных мест.

### Изменения в API

- При создании события (`POST /events`) теперь обязательно указывать поле `totalSeats` (должно быть > 0).
- При попытке забронировать место на заполненное мероприятие (`POST /events/{id}/book`) возвращается ошибка **409 Conflict**.

### Архитектура многопоточности

- **`SemaphoreSlim` в BookingService**: Защищает критическую секцию "проверка мест -> резервирование -> сохранение через `SaveChangesAsync`", гарантируя, что параллельные запросы не забронируют одно и то же место. Используется `static`, так как сервис Scoped, а синхронизация должна быть межзапросной.
- **`IServiceScopeFactory` в BookingBackgroundService**: Позволяет фоновым задачам получать отдельный `AppDbContext` для каждой обрабатываемой брони, избегая проблем с жизненным циклом Singleton-сервиса.
