# Event Manager API

Сервис для управления мероприятиями. Базовый каркас приложения, реализованный на ASP.NET Core Web API, с применением паттернов разделения ответственности, глобальной обработки ошибок и покрытием бизнес-логики юнит-тестами.

## Технологический стек

- **C# / .NET 8**
- **ASP.NET Core Web API**
- **In-memory хранилище** (для учебных целей)
- **Swagger (OpenAPI)** для документации
- **xUnit** для юнит-тестирования

## Архитектура проекта

Проект разработан с соблюдением принципов чистой архитектуры (Clean Architecture) и SOLID:

- **Разделение ответственности (SRP):** Логика строго разделена на инфраструктурный слой (доступ к данным) и доменный слой (бизнес-логика).
- **Инверсия зависимостей (DIP):** Доменные сервисы зависят от абстракций (`IEventStore`, `IBookingStore`), а не от конкретных реализаций в памяти.
- **Паттерн Repository:** Фильтрация, пагинация и прямое взаимодействие с коллекциями вынесены в классы `InMemoryEventStore` и `InMemoryBookingStore`.
- **Жизненные циклы DI:**
    - Хранилища (Stores) зарегистрированы как **Singleton**, так как содержат общее состояние приложения.
    - Доменные сервисы (Services) зарегистрированы как **Scoped**, так как они не имеют состояния и живут в рамках одного HTTP-запроса.
    - Фоновые задачи зарегистрированы как **Hosted** сервисы.
- **Обработка ошибок:** За 400 ошибки отвечает встроенный механизм валидации ASP.NET. За 404 и 500 ошибки отвечает кастомный `GlobalExceptionHandler`, использующий базовый класс `BaseApiException` для масштабируемости (OCP).
- **Observability (Логирование):** Внедрен `ILogger` в контроллеры и сервисы для трассировки жизненного цикла запросов и отладки.

## Запуск проекта

1. Клонируйте репозиторий и перейдите в папку решения:
    ```bash
    git clone https://github.com/dennisrotar/EventManagerAPI.git cd EventManagerAPI
    ```

2. Соберите проект:
    ```bash
    dotnet build
    ```
    
3. Запустите проект (обратите внимание на флаг `--project`, так как `.csproj` лежит во вложенной папке):
    ```bash
    dotnet run --project EventManager.Api/EventManagerAPI.csproj
    ```

4. Откройте браузер и перейдите по адресу Swagger UI (порт указывается в консоли при запуске, например `http://localhost:<порт>/swagger).

## Запуск тестов

Для запуска юнит-тестов выполните команду из корня решения:

```bash
dotnet test
```

## Документация API

### Базовые операции (CRUD)

|Метод|Эндпоинт|Описание|Успешный статус|
|---|---|---|---|
|`GET`|`/events`|Получить список мероприятий (с фильтрацией и пагинацией)|200 OK|
|`GET`|`/events/{id}`|Получить мероприятие по ID|200 OK|
|`POST`|`/events`|Создать новое мероприятие|201 Created|
|`PUT`|`/events/{id}`|Обновить мероприятие целиком|204 No Content|
|`DELETE`|`/events/{id}`|Удалить мероприятие|204 No Content|

### Фильтрация и пагинация (GET /events)

Эндпоинт `GET /events` принимает опциональные query-параметры для поиска:

- `title` (string) — поиск по названию (частичное совпадение, регистронезависимый).
- `from` (DateTime) — события, которые начинаются **не раньше** указанной даты.
- `to` (DateTime) — события, которые заканчиваются **не позже** указанной даты.
- `page` (int, по умолчанию 1) — номер страницы (не может быть < 1).
- `pageSize` (int, по умолчанию 10) — количество элементов на странице (не может быть < 1).

**Пример запроса:**

```text

GET /events?title=концерт&from=2024-01-01T00:00:00Z&page=2&pageSize=5
```

**Пример ответа пагинации:**

```json
{
  "totalCount": 12,
  "page": 2,
  "pageSize": 5,
  "items": [
    {
      "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
      "title": "Тренировка по баскету",
      "description": "Опоздаешь-отожмёшься!",
      "startAt": "2024-01-15T20:00:00Z",
      "endAt": "2024-01-15T23:00:00Z"
    }
  ]
}
```

## Правила валидации

Сервис осуществляет строгую проверку входных данных:

- **Title**: Не может быть пустой строкой.
- **Даты**: `EndAt` строго позже `StartAt`.
- **Дата в прошлом**: `StartAt` не может быть меньше текущего времени сервера (UTC).
- **Пагинация**: Параметры `page` и `pageSize` не могут быть меньше 1 (возвращает 400 Bad Request).

## Обработка ошибок

Все ошибки (как от встроенного валидатора, так и от бизнес-логики) возвращаются в **единообразном формате RFC 7807 (Problem Details)**.

### Пример 1: Ошибка валидации (400 Bad Request)

Возникает при невалидных данных в теле запроса или некорректных параметрах пагинации.


```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "instance": "/events",
  "traceId": "00-873ba4af2ca7f9c6f861fb3cdb5c6669-0c1015a54df57858-00",
  "errors": {
    "Title": [
      "Название мероприятия не может быть пустым!"
    ]
  }
}
```


### Пример 2: Ресурс не найден (404 Not Found)

Возникает при запросе, обновлении или удалении несуществующего ID.


```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.5",
  "title": "Not Found",
  "status": 404,
  "detail": "Мероприятие с ID 00000000-0000-0000-0000-000000000000 не найдено",
  "instance": "/events/00000000-0000-0000-0000-000000000000",
  "traceId": "00-2f74d953e90ca3678214d2e930e48b51-fd9e4692eb25c93b-00"
}
```

## Бронирование мероприятий (Отложенная обработка)

Сервис поддерживает паттерн " быстрый ответ + отложенная обработка" для бронирования.

### Эндпоинты

|Метод|Эндпоинт|Описание|
|:--|:--|:--|
|`POST`|`/events/{id}/book`|Создать бронь (Возвращает 202 Accepted)|
|`GET`|`/bookings/{id}`|Получить статус брони|

### Модель Booking

- `Pending` — создана, ожидает обработки фоновым сервисом.
- `Confirmed` — успешно обработана.
- `Rejected` — отклонена.

### Как это работает

1. Запрос `POST /events/{id}/book` мгновенно сохраняет бронь со статусом `Pending` и возвращает `202 Accepted` с заголовком `Location`.
2. Фоновый сервис `BookingBackgroundService` постоянно опрашивает базу в памяти.
3. Найдя `Pending` бронь, сервис "имитирует" обращение к платежной системе (задержка 2 секунды) и меняет статус на `Confirmed`.
4. При последующем `GET /bookings/{id}` вы увидите актуальный статус.
